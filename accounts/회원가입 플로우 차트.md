# 회원가입 플로우 차트

1. 가입
2. 정보 기입 페이지 (이름, 이메일, 비밀번호, 약관동의)
3. 추가 정보 (나이[밝히고 싶지 않음 가능], 유형[중고교생/학부생/일반])
4. 미활성 상태로 저장 (로그인 불가능 상태)

## 이메일 최초 인증
5. 이메일 미인증 상태로 로그인 시도시 이메일 인증 페이지로 이동
6. 일회성 활성화 링크 생성
7. 이메일 발송
8. 유저가 메일함에서 링크를 클릭하면 서버가 유효성을 검증하고, 유효하다면 유저의 상태를 활성화로 변경. 이제 정상 로그인 가능.

**주의점 : 보안 취약점에 대해서 생각하고 이에 대응해**


(보안 취약점)

### 1. [중요] 로그인 시 패스워드 검증 타이밍 어택 (Timing Attack) 취약점                                              
                                                                                                                        
  • 상세:                                                                                                               
  현재 로그인 뷰(auth.py)에서 미인증 유저 여부를 판별하기 위해 다음과 같이 코드가 실행됩니다.                    
      1.  authenticate(username=email, password=password)  호출 ➡️ (유저가 inactive하므로 내부적으로 해시 검증 후  None  
      반환)                                                                                                             
      2.  else  블록 진입 후  inactive_user.check_password(password)  호출 ➡️ (해시 검증을 한 번 더 실행)                
  • 보안 위협:                                                                                                          
  미인증 유저가 로그인 시도를 할 때 패스워드 해싱 함수(PBKDF2/bcrypt)가 2회 연속 실행됩니다.                            
  일반 유저 로그인(해싱 1회, 약 100ms) 대비 미인증 유저 로그인(해싱 2회, 약 200ms)의 응답 시간이 2배 가까이 길어지기    
  때문에, 공격자는 응답 지연 시간 차이를 분석해 **"특정 이메일이 가입되어 있으며 미인증 상태인지"를 원격으로 구분(유동적
  타이밍 어택)**할 수 있습니다.                                                                                         
  ──────                                                                                                                
  ### 2. [중요] 호스트 헤더 인젝션 (Host Header Injection) 위험                                                         
                                                                                                                        
  • 상세:                                                                                                               
  email_verification.py에서 메일 본문에 삽입할 인증 링크 URL을 생성할 때,  .env 에  SITE_URL 이 설정되어 있지 않은 경우       
  fallback으로  request.build_absolute_uri() 를 사용합니다.                                                             
  • 보안 위협:                                                                                                          
  만약 운영 환경 배포 시  settings.py 의  ALLOWED_HOSTS 가 와일드카드( * )로 열려 있고  .env 에  SITE_URL 을 설정하지   
  않았다면, 공격자가 HTTP 요청의  Host  헤더를 자기 서버의 도메인(예:  attacker.com )으로 조작하여 로그인을 시도할 수   
  있습니다.                                                                                                             
  이 경우 메일 서버는 공격자 도메인이 포함된 인증 링크( http://attacker.com/accounts/verify/uid/token/ )를 사용자에게   
  보내게 되고, 무심코 링크를 클릭한 사용자의 인증 토큰이 공격자 서버로 유출(Hijacking)될 수 있습니다.                   
  ──────                                                                                                                
  ### 3. [보통] 다중 프로세스(Gunicorn/uWSGI) 환경에서 Rate Limit 우회 가능성                                           
                                                                                                                        
  • 상세:                                                                                                               
  메일 발송 제한을 위해  django.core.cache 를 사용하여 1분 제한 및 IP 누적 카운트를 체크합니다.                         
  • 보안 위협:                                                                                                          
  Django의 기본 캐시 백엔드가 로컬 메모리 캐시( LocMemCache )로 구성된 경우, 서버 인스턴스가 여러 개이거나 다중 워커    
  프로세스(Gunicorn Workers)로 기동 중일 때 각 프로세스끼리 메모리가 공유되지 않습니다.
  따라서 공격자가 요청을 분산하여 계속 찌를 경우, 각 프로세스의 메모리 임계치만큼 제한을 초과해서 인증 메일 발송 API를  
  남용(Spamming)할 수 있습니다.
  (이 취약점은 배포 환경에서 Redis나 Database 캐시 같은 중앙 집중형 캐시 서버를 연동함으로써 해결이 가능합니다.)        
  ──────
  ### 4. [보통] 세션 잔류 데이터 미정리 (Session Pollution)
  
  • 상세:
  로그인 실패 혹은 회원가입 후 세션에 저장하는  request.session['unverified_email'] 은 사용자가 메일 링크를 클릭해      
  계정을 활성화(verification.py)한 이후에도 명시적으로 세션에서 삭제( pop  또는  del )되지 않고 계속 남아있습니다.       
  • 보안 위협:
  직접적인 침투 취약점은 아니지만 세션 오염(Session Pollution)에 해당하며, 인증이 끝난 이후에도 브라우저 세션 쿠키가    
  탈취당할 경우 세션 내부에 존재하지 않아도 되는 사용자의 이메일 주소가 공격자에게 PII(개인정보)로 추가 유출되는        
  리스크가 존재합니다.
  ──────
  ### 5. [낮음] 인증 메일 링크의 유효시간 설정 누락
  
  • 상세:
  현재 인증 링크 토큰은 Django의 패스워드 재설정 만료 규정( PASSWORD_RESET_TIMEOUT )을 공용으로 상속받고 있습니다.      
  • 보안 위협:
   settings.py 에 별도로  PASSWORD_RESET_TIMEOUT  값을 재정의하지 않았다면 Django 기본값인 3일(72시간) 동안 토큰이      
  유효하게 살아있습니다. 이메일 최초 인증의 유효기간치고는 3일은 지나치게 긴 편이므로, 토큰 탈취나 오용 가능성을 낮추기 
  위해 유효시간을 더 짧게(예: 1시간~24시간) 제한하는 것이 안전합니다.
